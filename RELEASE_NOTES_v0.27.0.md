# NeuralMind v0.27.0 — multi-language: Rust (no graphify)

**Release Date:** June 2026

## TL;DR

v0.16.0 took the built-in tree-sitter backend multi-language with TypeScript
and Go. v0.27.0 adds **Rust** behind the same seam:

```bash
pip install neuralmind
neuralmind build .        # now indexes Python, TypeScript, Go, AND Rust — bundled
```

Rust (`.rs`) is registered through the same `_SUFFIX_LANG` → `_EXTRACTORS`
seam, emitting the same graphify-compatible `graph.json` the whole retrieval
stack already consumes. Nothing downstream changed — only the set of languages
the *producer* understands. The bundled backend now covers **four languages**.

As always, it's **proven at parity, not asserted**: the backend parity gate now
checks Rust too (100% symbol coverage on the reference fixture, zero dangling
edges).

## What the agent actually sees, post-install

On a Rust crate with no `graphify-out/graph.json`, `neuralmind build` now
**succeeds on its own** and prints:

```
[neuralmind] generated code graph via the built-in tree-sitter backend → graphify-out/graph.json
```

The agent then gets the same L0–L3 progressive disclosure, synapse layer, graph
view, and MCP tools — now over a Rust codebase. A mixed-language repo
(Python + TS + Go + Rust) is indexed in one pass: each file is dispatched to its
language's extractor and merged into a single graph. Rust's `target/` build
directory is skipped automatically, like `node_modules` / `dist` / `build`.

## How the Rust extractor maps onto the model

`neuralmind/graphgen.py` dispatches each `.rs` file to the Rust extractor, which
maps the grammar's node types onto the **same** node/edge model every other
language uses:

| | Rust |
|---|---|
| functions | `function_item` (free fns, `impl` methods, `trait` signatures) |
| types | `struct_item` / `enum_item` / `union_item` / `trait_item` / `type_item` |
| symbols | struct fields, enum variants, `const`/`static` items |
| imports | `use` paths, resolved against a module-key table (`imports_from`) |
| inherits | `impl Trait for Type` → `inherits` edge `Type → Trait` |
| rationale | leading doc comments (`///`, `//!`, `/** */`) |
| calls | `call_expression`, last path segment matched by name (best-effort) |

`impl` methods are attached to their type node, so `Connection::get_connection`
is `contained` by `Connection`. `use crate::db::connection::Connection`
resolves to an `imports_from` edge to `db/connection.rs`. Every Rust file emits
the same `code` / `rationale` / `document` nodes and `contains` /
`imports_from` / `inherits` / `calls` / `rationale_for` edges, plus the same
balanced per-file communities — nothing downstream of `graph.json` changed.

`tree-sitter-rust` is now a **core dependency**, so the language is bundled. A
grammar that somehow isn't importable is skipped per-file (Python stays the only
hard requirement), so a build never fails for a missing optional grammar.

## The parity gate, now with Rust

For Python the gate runs the full **faithfulness A/B + reduction** (a gold-fact
set exists). TypeScript, Go, and Rust have no gold-fact set, so the gate proves
parity **structurally**: the built-in backend must recover at least **90%** of
the symbols in the committed gold graph, with a valid, dangling-free graph. On
the reference fixture it recovers **100%**:

| Language | gold symbols | built-in covers | dangling |
|---|---:|---:|---:|
| TypeScript | 54 | 54 (100%) | 0 |
| Go | 45 | 45 (100%) | 0 |
| Rust | 49 | 49 (100%) | 0 |

The floor is overridable via `NEURALMIND_PARITY_COVERAGE_FLOOR`.

## Honest scope & caveats

- **graphify can't parse Rust**, so unlike TS/Go (whose gold was produced by
  graphify) the Rust gold is generated by the built-in extractor and reshaped
  into graphify's schema. The **independent** correctness oracle is the
  hand-listed expected symbols/edges in `tests/test_graphgen.py`
  (`RustTests`); the parity gate then guards against regressions.
- **Heuristic call/inherit edges**, as in every language — best-effort by name,
  no type/scope resolution. Macros (`macro_rules!`, `println!`) and `cfg!`
  variants are not expanded (the same static-structure limitation every
  tree-sitter backend has). The optional SCIP precision pass
  (`NEURALMIND_PRECISION=1`) remains the path to exact edges.
- **graphify still takes priority** where present, in every language.

## What ships

- **`neuralmind/graphgen.py`** — Rust extractor (`_rust_extract_symbols` /
  `_rust_resolve_edges`) registered in `_EXTRACTORS`; `.rs` in
  `SUPPORTED_SUFFIXES`; `target/` added to the default ignores (mirrored in
  `neuralmind/watcher.py`).
- **`evals/parity/run.py`** — Rust wired into the per-language structural-
  coverage gate.
- **`tests/fixtures/sample_project_rust/`** — a Rust crate mirroring the
  Python/Go fixtures, with a committed gold graph and a mirrored
  `benchmark_queries_rust.json`.
- `tree-sitter-rust` added as a core dependency.
- Tests: `RustTests` extraction class in `tests/test_graphgen.py`; Rust wired
  into `tests/test_polyglot_fixtures.py`.
- **`docs/prd/language-extractor-expansion.md`** — the PRD this release
  executes Phase 1 of.

## Upgrade

```bash
pip install --upgrade neuralmind
```

No migration, no config changes. Existing setups are unchanged. On a fresh Rust
repo you can now skip graphify entirely:

```bash
pip install neuralmind
neuralmind build /path/to/your/rust-crate
neuralmind query "how does authentication work?" /path/to/your/rust-crate
```
