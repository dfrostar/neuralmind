# NeuralMind v0.28.0 — multi-language: Java (no graphify)

**Release Date:** June 2026

## TL;DR

v0.27.0 added Rust. v0.28.0 adds **Java** behind the same seam, taking the
bundled tree-sitter backend to **five languages**:

```bash
pip install neuralmind
neuralmind build .   # now indexes Python, TypeScript, Go, Rust, AND Java — bundled
```

Java (`.java`) is registered through the same `_SUFFIX_LANG` → `_EXTRACTORS`
seam, emitting the same graphify-compatible `graph.json` the whole retrieval
stack already consumes. Nothing downstream of `graph.json` changed — only the
set of languages the *producer* understands. Proven at parity by the CI gate
(100% symbol coverage on the reference fixture, zero dangling edges).

## What the agent actually sees, post-install

On a Java project (Maven/Gradle/plain) with no `graphify-out/graph.json`,
`neuralmind build` now **succeeds on its own** and prints:

```
[neuralmind] generated code graph via the built-in tree-sitter backend → graphify-out/graph.json
```

The agent then gets the same L0–L3 progressive disclosure, synapse layer, graph
view, and MCP tools — now over a Java codebase. A mixed-language repo
(e.g. a Java service with a Rust sidecar and Python tooling) is indexed in one
pass: each file is dispatched to its language's extractor and merged into a
single graph.

## How the Java extractor maps onto the model

| | Java |
|---|---|
| types | `class` / `interface` / `enum` / `record` / `annotation` declarations |
| methods | `method_declaration`, `constructor_declaration` (contained by their type) |
| symbols | `field_declaration` variables, `enum_constant`s |
| imports | `import com.foo.Bar;` resolved by fully-qualified name → `imports_from` |
| inherits | `extends` / `implements` → `inherits` edge (external bases synthesized) |
| rationale | leading Javadoc (`/** … */`) and line comments |
| calls | `method_invocation`, matched by method name (best-effort) |

Each top-level type registers both its **fully-qualified name**
(`com.example.db.Connection`) and its **simple name** in the module-key table,
so an `import com.example.db.Connection;` in another file resolves to an
`imports_from` edge. Nested types are contained by their outer type. Node
provenance is anchored on the identifier's line, so annotations/modifiers
(`@Override`, `public static`) never push a symbol's `source_location` onto the
wrong line.

`tree-sitter-java` is now a **core dependency**, so the language is bundled. A
grammar that somehow isn't importable is skipped per-file (Python stays the only
hard requirement), so a build never fails for a missing optional grammar.

## The parity gate, now with Java

graphify can't parse Java, so the gate proves parity **structurally**: the
built-in backend must recover ≥90% of the symbols in the committed gold graph
with a valid, dangling-free graph. On the reference fixture it recovers
**100%**:

| Language | gold symbols | built-in covers | dangling |
|---|---:|---:|---:|
| TypeScript | 54 | 54 (100%) | 0 |
| Go | 45 | 45 (100%) | 0 |
| Rust | 49 | 49 (100%) | 0 |
| Java | 54 | 54 (100%) | 0 |

## Honest scope & caveats

- The Java gold is generated by the built-in extractor (graphify can't parse
  Java) and reshaped into graphify's schema; the **independent** correctness
  oracle is the hand-listed symbols/edges in `tests/test_graphgen.py`
  (`JavaTests`). The parity gate then guards against regressions.
- **Heuristic call/inherit edges**, as in every language — best-effort by name,
  no overload/type resolution. **`record` components** are not yet extracted as
  field symbols (the record itself is captured as a type). Annotations are not
  represented as edges. The optional SCIP precision pass remains the path to
  exact edges.
- **graphify still takes priority** where present, in every language.

## What ships

- **`neuralmind/graphgen.py`** — Java extractor (`_java_extract_symbols` /
  `_java_resolve_edges`) registered in `_EXTRACTORS`; `.java` in
  `SUPPORTED_SUFFIXES`.
- **`evals/parity/run.py`** — Java wired into the per-language structural-
  coverage gate.
- **`tests/fixtures/sample_project_java/`** — a Java project mirroring the
  Python/Go/Rust fixtures, with a committed gold graph and a mirrored
  `benchmark_queries_java.json`.
- `tree-sitter-java` added as a core dependency.
- Tests: `JavaTests` in `tests/test_graphgen.py`; Java wired into
  `tests/test_polyglot_fixtures.py`.

## What's next

Per the language-extractor PRD, the gated recipe continues: **C/C++**
(structural-first, the macro-heavy hard case) and **Ruby** — one language per
PR, each clearing the same parity gate.

## Upgrade

```bash
pip install --upgrade neuralmind
```

No migration, no config changes. On a fresh Java repo you can now skip graphify
entirely:

```bash
pip install neuralmind
neuralmind build /path/to/your/java-project
neuralmind query "how does authentication work?" /path/to/your/java-project
```
